# 📔 Journal — оффлайн-дневник в формате «Telegram с самим собой»

## 🧠 Общая идея проекта

Разработать **Android-приложение** для оффлайн-ведения заметок, журналов и личных логов.

### Главная концепция

- приложение ощущается как **«Telegram с самим собой**»;
- пользователь ведёт **тематические журналы**;
- записи создаются в **формате сообщений**;
- **время сообщений нельзя изменять**;
- всё работает полностью **локально без сервера**;
- максимальный акцент на **UX, анимациях, tactile feedback, звуках и эмоциональном опыте** использования.

### Эмоциональный тон приложения

Приложение должно создавать ощущение:

- ✨ **премиальности**;
- 🧠 **продуманности**;
- 🧬 **«живого» интерфейса**;
- 🤝 **глубокого взаимодействия** с пользователем.

### 6.2.1 Голосовые заметки

#### Описание

Запись голоса — обязательная функция MVP. Пользователь может создать голосовое сообщение прямо из чата журнала, аналогично голосовым в Telegram.

#### Возможности

- запись аудио с микрофона;
- визуализация уровня звука в реальном времени (waveform);
- отправка / отмена записи;
- воспроизведение внутри чата;
- отображение длительности сообщения.

#### Требования

- запись хранится локально в формате AAC / MP3;
- файл привязывается к сообщению в SQLite;
- максимальная длительность записи — без ограничений (или до 30 минут);
- мгновенный старт записи (без задержек);
- haptic feedback при начале и завершении записи.

### 6.4 Напоминания

#### Тип 1: Будильник

**Концепция:** уведомление работает как будильник — не исчезает, пока пользователь не примет действие. Подходит для важных напоминаний, которые нельзя пропустить.

**Особенности:**

- полноэкранный экран уведомления;
- звук играет непрерывно до реакции пользователя;
- вибрация с нарастающей интенсивностью;
- блокировка интерфейса до взаимодействия;
- работает даже при заблокированном экране (full-screen intent).

**Действия:**

- отложить (snooze) — с настраиваемым интервалом;
- выключить (dismiss);
- открыть связанную заметку / журнал.

**Настройки:**

- выбор мелодии;
- настройка громкости;
- snooze-интервал (5 / 10 / 15 / 30 минут);
- максимальная длительность звука (по умолчанию — до реакции, максимум 5 минут непрерывного звука).

#### Примеры использования

- напоминание о тренировке;
- дедлайн по задаче;
- важное событие в журнале.

#### Тип 2: Push-уведомление

- локальное уведомление;
- тихий режим;
- вибрация;
- deep-link в заметку.

### 6.10 Google Drive Sync *(optional)*

**Назначение** — резервное копирование и перенос данных между устройствами.

**Концепция** — как резервное копирование WhatsApp: пользователь вручную включает синхронизацию, данные загружаются в Google Drive и восстанавливаются на другом устройстве.

**Особенности:**

- не постоянная синхронизация — только по запросу пользователя;

**Настройки:**

- привязка Google-аккаунта;
- автобэкап (опционально): ежедневно / еженедельно / вручную;
- шифрование бэкапа (пароль);

**Ограничения:**

### 6.11 Dynamic Island *(Android)*

**Концепция** — приложение поддерживает современные Android-компоненты для отображения активных процессов поверх интерфейса, аналогично Dynamic Island на iOS.

**Реализация:**

- **Live Notifications** (Android 13+): отображение текущих напоминаний и таймеров в статус-баре;
- **Heads-up notifications**: всплывающие уведомления для будильников;
- **Always-on Display** *(опционально)*: отображение ближайшего напоминания на экране блокировки.

**Примеры:**

- активный будильник — индикатор в статус-баре;
- идёт запись голоса — индикатор записи;
- циклическое напоминание — pulsing notification dot.

**Технологии:**

| Задача                    | Решение                              |
| ------------------------- | ------------------------------------ |
| Live Notifications        | Android NotificationCompat + Foreground Service |
| Heads-up                  | Android NotificationChannel (high priority) |
| Always-on Display         | Android Widget / Complications API   |

## 🎨 UX/UI концепция

### 7.1 Главный принцип

Приложение должно ощущаться:

- дорогим;
- живым;
- tactile;
- fluid.

При этом **визуальный слой намеренно сдержанный** (см. §7.1.2): ощущение **премиальности** и глубины достигается **анимацией, физикой взаимодействия, тактильным и звуковым откликом**, а не перегрузкой цветом и декоративными контролами.

### 7.1.1 Иконка приложения

**Требование:** иконка приложения — монохромная.

**Концепция:**

- один цвет (или оттенки одного цвета);
- минималистичный дизайн;
- узнаваемость на любом фоне;
- соответствие стилю Material You (Android 12+).

**Адаптивность:**

- поддержка adaptive icon (Android 8+);
- автоматическая подстройка под системную палитру (Material You dynamic color);
- отдельные версии для тёмной и светлой темы.

**Форматы:**

| Тип          | Размер        |
| ------------ | ------------- |
| Adaptive     | 108×108 dp    |
| Legacy       | 48×48 dp      |
| Round        | 108×108 dp    |
| Foreground   | 66×66 dp (внутри adaptive) |

### 7.1.2 Визуальный язык, палитра и скорость задачи

**Роль дизайна в продукте:** интерфейс трактуется как **инструмент быстрой фиксации** мыслей и напоминаний. Визуальная система должна снижать когнитивную нагрузку и время принятия решения, а не конкурировать с контентом за внимание.

**Палитра и иерархия**

- опираемся на **узкую палитру — по сути 2–3 семантических цвета** в типичном состоянии экрана (фон поверхности, основной текст и действия, при необходимости один акцент или системный цвет состояния — успех, ошибка, отключено);
- **базовый светлый режим по умолчанию:** светлый/белый фон, **основные действия и ключевой текст — тёмные (в т.ч. чёрные)** для максимального контраста и предсказуемости без «украшательств»;
- тёмная тема и адаптация под Material You **не отменяют** принципа малого числа семантических цветов: системные динамические оттенки используются дисциплинированно, без размывания ролей «фон / текст / действие».

**Контролы и «хром» интерфейса**

- **первичные действия** по возможности оформляются как **текстовые кнопки (text / label buttons)** без декоративной обводки: пользователь считывает намерение по подписи и контексту, а не по «рамке»;
- отсутствие border у типовых кнопок — осознанный выбор в пользу **скорости сканирования** и визуальной тишины; при необходимости границы и заливки используются для **вторичных** действий или редких состояний (disabled, outline-secondary), чтобы не дублировать визуальный вес всего экрана;
- **иконки** допускаются **выборочно**: там, где они заменяют длинную подпись, дают мгновенное узнавание (навигация, системные действия) или поддерживают международную аудиторию; массовое заполнение экрана иконками без текстовой опоры не является целевым паттерном.

**Дата, время и напоминания**

- сценарий **«записать напоминание с датой/временем»** — **критический путь UX**: требуется **оригинальный, но предельно быстрый** способ ввода даты и времени (комбинация умных пресетов, компактного picker, подсказок по контексту, опционально естественного ввода — конкретная механика задаётся отдельной проработкой);
- целевой ориентир — **минимум касаний и экранов** от намерения до подтверждённого напоминания без потери точности и без forcing-потоков, которые заставляют пользователя «воевать» с календарём.

**Согласование с остальной концепцией**

- «Живость» и fluid из §7.1 **не противоречат** минимализму хрома: микровзаимодействия, переходы и пружины усиливают ощущение качества **без** усложнения статической картинки;
- при конфликте приоритетов между **украшением статического UI** и **сокращением пути пользователя** выбирается второе.

#### Permissions (Scopes)

Использовать **только** scope:

`https://www.googleapis.com/auth/drive.appdata`

Это даёт доступ **только** к скрытой папке приложения в Google Drive (appDataFolder).

### Google Drive Backup

#### Назначение

Приложение должно сохранять **backup пользовательских данных** в Google Drive пользователя.

#### Требования

- backup хранится в **appDataFolder**;
- файлы backup **скрыты от пользователя** в интерфейсе Drive;
- приложение **не имеет доступа** к остальным файлам Google Drive пользователя;
- backup **привязан** к Google-аккаунту пользователя.

#### Функции backup

Приложение должно поддерживать:

- создание backup;
- обновление backup;
- удаление старого backup;
- восстановление backup;
- автоматическую синхронизацию.

#### Формат хранения

Backup хранится в формате **JSON**.

Пример:

```json
{
  "version": 1,
  "updatedAt": 1710000000,
  "data": {}
}
```

#### Автоматическая синхронизация

Приложение должно:

- автоматически **загружать** backup после изменений;
- **проверять** наличие backup при запуске;
- **восстанавливать** данные после входа пользователя.

#### Безопасность

**Требования:**

- использовать только **OAuth-авторизацию** Google;
- **не хранить** пароли пользователя;
- хранить **access token** безопасно (защищённое хранилище ОС / best practices платформы);
- использовать **HTTPS** для запросов;
- **не использовать** полный доступ к Google Drive (только `drive.appdata`).

#### Offline-поддержка

Приложение должно:

- работать **без интернета** (локальные операции и очередь изменений);
- выполнять **синхронизацию** после подключения к сети.

#### Обработка ошибок

Необходимо обрабатывать:

- отсутствие интернета;
- ошибки авторизации;
- истёкший access token (обновление токена / повторная авторизация);
- ошибки Google Drive API;
- отсутствие backup-файла;
- повреждённый backup (невалидный JSON, несовместимая `version`, и т. п.).

CREATE TABLE journals (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,

  FOREIGN KEY (journal_id) REFERENCES journals(id) ON DELETE CASCADE

CREATE TABLE general_notes (
  id TEXT PRIMARY KEY,
  title TEXT,
