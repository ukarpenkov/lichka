# Development and Release Rules

## 1. Архитектура — Feature-Sliced Design (FSD)

Проект использует архитектуру **Feature-Sliced Design** (FSD). Это методология фронтенд-архитектуры, которая обеспечивает чёткую структуру кода и масштабируемость.

### Слои (Layers)


| Слой         | Назначение                                      | Примеры                     |
| ------------ | ----------------------------------------------- | --------------------------- |
| **app**      | Глобальные настройки, провайдеры, инициализация | App.tsx, навигация, темы    |
| **pages**    | Страницы/экраны приложения                      | HomeScreen, SettingsScreen  |
| **widgets**  | Самостоятельные блоки UI, состоящие из фич      | Header, Sidebar             |
| **features** | Пользовательские действия и бизнес-логика       | login, addNote, syncData    |
| **entities** | Бизнес-сущности                                 | User, Note, Category        |
| **shared**   | Переиспользуемый код, UI-компоненты, утилиты    | Button, api-client, helpers |


### Принципы

- **Направление зависимостей**: каждый слой может импортировать только из нижележащих слоёв (app → pages → widgets → features → entities → shared)
- **Запрещены циклические зависимости** между слайсами одного уровня
- **Публичное API**: каждый слайс экспортирует только через `index.ts` (public API), внутренняя реализация скрыта
- **Низкая связанность, высокая сцепленность**: каждый модуль отвечает за одну задачу

## 2. Коммиты

### Обязательные правила

- **Каждое изменение кода должно сопровождаться коммитом.** Не допускается работа без коммитов или накопление больших незафиксированных изменений.
- Коммиты должны быть **атомарными** — один коммит = одно логическое изменение.
- Используй формат **Conventional Commits**:

```
<type>(<scope>): <description>

[optional body]
[optional footer]
```

### Типы коммитов


| Тип        | Описание                                      |
| ---------- | --------------------------------------------- |
| `feat`     | Новая функциональность                        |
| `fix`      | Исправление бага                              |
| `docs`     | Изменения в документации                      |
| `style`    | Форматирование, точки с запятой и т.д.        |
| `refactor` | Рефакторинг без изменения функциональности    |
| `test`     | Добавление или изменение тестов               |
| `chore`    | Изменения в конфигурации, зависимостях и т.д. |


### Примеры

```
feat(notes): add voice note recording
fix(sync): resolve Google Drive auth timeout
docs: add FSD architecture rules
test(entities/note): add unit tests for Note model
```

### Авторство

- В коммитах **нет** `Co-Authored-By: Claude…` / AI trailers — только автор-человек
- Claude Code: `.claude/settings.json` → `"attribution": { "commit": "", "pr": "" }`
- Git hook: `.githooks/commit-msg` (включается через `npm install` / `npm run prepare`)

## 3. Документация обсуждений и репортов

### Обсуждение новых фич

Перед внедрением новой фичи **обязательно** создаётся документ-обсуждение в `docs/features/`:

```
docs/features/<feature-name>-proposal.md
```

Документ должен содержать:

- **Название фичи**
- **Описание проблемы** — какую задачу решает
- **Предлагаемое решение** — техническое описание реализации
- **Влияние на архитектуру** — какие слои FSD затрагивает
- **Альтернативы** — что ещё рассматривалось и почему выбран именно этот вариант
- **Оценка сложности** — время и риски
- **Статус** — draft / approved / rejected / implemented

### Репорты по коду

После завершения имплементации фичи создаётся репорт в `docs/reports/`:

```
docs/reports/<feature-name>-report.md
```

Репорт должен содержать:

- **Что сделано** — описание реализованной функциональности
- **Структура изменений** — список файлов и модулей, которые были изменены или созданы
- **Принятые решения** — неочевидные технические решения и их обоснование
- **Известные ограничения** — что пока не реализовано или работает с ограничениями
- **Тестирование** — какие тесты написаны, какие сценарии покрыты

## 4. Покрытие тестами

### Требования

- **Каждая новая фича должна иметь тесты.** Код без тестов не мержится в основную ветку.
- Минимальное покрытие: **80%** по каждому модулю.
- Используются следующие типы тестов:


| Тип             | Что тестирует                      | Инструмент                   | Расположение                                 |
| --------------- | ---------------------------------- | ---------------------------- | -------------------------------------------- |
| **Unit**        | Отдельные функции, модели, утилиты | Jest                         | `__tests__/` или `*.test.ts` рядом с модулем |
| **Integration** | Взаимодействие между модулями      | Jest + React Testing Library | `features/<name>/__tests__/`                 |
| **E2E**         | Конечные сценарии пользователя     | Detox / Maestro              | `e2e/`                                       |


### Правила написания тестов

- Тесты должны быть **детерминированными** — один и тот же результат каждого запуска
- Один тест = один сценарий (Arrange → Act → Assert)
- Используй **моки** для внешних сервисов (API, Google Drive, файловая система)
- Тесты должны быть **независимыми** друг от друга
- Названия тестов описывают поведение, а не реализацию:

```typescript
// ✅ Правильно
it('should save note to local storage when offline')

// ❌ Неправильно
it('should call AsyncStorage.setItem')
```

### CI/CD

- Тесты запускаются автоматически перед каждым мержем
- Пул-реквест не мержится, если хотя бы один тест не проходит
- Отчёт о покрытии генерируется автоматически
