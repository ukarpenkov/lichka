# Корневой рефакторинг обработки клавиатуры в чате

**Дата:** 2026-07-03
**Промпт/задача:** Баг «блок отправки сообщения зависает в середине экрана при открытии чата». После фикса — пройтись по всему приложению, убрать костыли и сделать обработку клавиатуры правильно, чтобы подобные ошибки не повторялись.

## Контекст
Баг с положением блока ввода / зазором над клавиатурой чинили многократно (см. историю коммитов и `docs/bugs/*keyboard*`, `docs/bugs/chat-composer-stuck-mid-screen.md`): переключали `adjustResize` ↔ `adjustNothing`, пробовали `KeyboardAvoidingView`, добавляли ручной `useAnimatedKeyboard` + `paddingBottom`, затем JS-флаг `keyboardVisible` через `Keyboard.addListener`. Это классический whack-a-mole: каждый фикс порождал новую регрессию.

## Корневая причина
1. Ручная компенсация высоты клавиатуры была **продублирована** в `ChatRoomScreen` и `MessageComposer` и легко рассинхронизировалась.
2. Видимость клавиатуры определялась **JS-флагом** (`keyboardVisible`), который обновлялся `Keyboard.addListener('keyboardDidShow/Hide')`. На Android при `windowSoftInputMode="adjustNothing"` эти события нестабильны и гоняются с монтированием экрана.
3. `useAnimatedKeyboard().height` при навигации кратковременно оставался ненулевым при уже закрытой клавиатуре → `paddingBottom` сдвигал контент вверх → блок ввода оказывался посреди экрана.

## Что сделано
- Создан единый хук `useKeyboardHeight()` в `src/shared/lib/keyboard.ts` — **единый источник правды** по высоте клавиатуры.
- Хук гейтит высоту по `KeyboardState` из reanimated (`OPEN`/`OPENING`) на **UI-потоке**: эффективная высота ненулевая только когда клавиатура реально открыта/открывается, иначе `0`. Это делает «зависание посреди экрана» невозможным.
- Из `ChatRoomScreen` и `MessageComposer` удалены JS-листенеры `Keyboard.addListener`, флаги `keyboardVisible` и прямые вызовы `useAnimatedKeyboard`.
- Оба компонента теперь потребляют один и тот же хук; дублирования логики больше нет.
- Обновлён мок reanimated в `jest.setup.js` (добавлены `state`, `useDerivedValue`, `KeyboardState`).

## Изменённые файлы
- `src/shared/lib/keyboard.ts` — новый хук `useKeyboardHeight()` (UI-thread, гейт по `KeyboardState`)
- `src/shared/lib/index.ts` — публичный экспорт хука
- `src/pages/chat-room/ChatRoomScreen.tsx` — переход на `useKeyboardHeight`, удалены листенеры и `keyboardVisible`, упрощён `chatAreaAnimatedStyle`
- `src/widgets/message-composer/MessageComposer.tsx` — переход на `useKeyboardHeight`, удалены листенеры и `keyboardVisible`
- `jest.setup.js` — мок reanimated дополнен `state`, `useDerivedValue`, `KeyboardState`
- `docs/bugs/chat-composer-stuck-mid-screen.md` — обновлён раздел причины/исправления

## Принятые решения
- **UI-thread гейт по `KeyboardState` вместо JS-флага.** Никаких `Keyboard.addListener`, никаких гонок с навигацией — состояние читается там же, где и высота (на UI-потоке).
- **Один общий хук вместо дублирования.** Это устраняет корневую причину рассинхрона между экраном чата и композером.
- **Базовый layout гарантирует прижатие ко дну.** `chatArea` (`flex: 1`) → `FlatList` (`flex: 1`) + `MessageComposer`; при нулевой высоте клавиатуры композер всегда внизу.
- **`adjustNothing` в манифесте сохранён.** Ручная reanimated-компенсация теперь надёжна, поэтому нативный resize (нестабильный из-за `react-native-screens`) не нужен.
- **Прочие места с текстовым вводом не трогали.** `MessageEditor` и `PeriodPicker` используют стандартный `KeyboardAvoidingView` в центрированных модалках — это не костыль; они не участвовали в баге.

## Известные ограничения
- В момент закрытия клавиатуры (`KeyboardState.CLOSING`) эффективная высота сразу становится `0`, поэтому композер мгновенно «падает» вниз, пока клавиатура доезжает анимацией. Визуально незаметно и заведомо лучше, чем зависание посреди экрана.
- iOS: ручная компенсация в `chatAreaAnimatedStyle` по-прежнему только для Android (как и раньше); на iOS подъём делает система.
- Изменения затрагивают JS-логику; нативная пересборка Android не требуется. Проверка на устройстве всё же рекомендована.

- Ручные сценарии для проверки на устройстве:
  - Открытие чата «Saved messages» без предварительной клавиатуры — композер прижат ко дну.
  - Открытие чата сразу после набора текста на другом экране — композер прижат ко дну (нет гонки).
  - Тап по полю ввода — композер поднимается над клавиатурой без зазора.
  - Скрытие клавиатуры — композер возвращается ко дну.
