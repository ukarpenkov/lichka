# Исправление будильника: надёжность при закрытом приложении и сообщение в чате

**Дата:** 2026-07-03
**Промпт/задача:** Будильник не работает при закрытом приложении; сообщение с текстом будильника не появляется в чате

## Что сделано

### Баг 1: Будильник не срабатывает при закрытом/свёрнутом приложении

**Причина:** `scheduleAlarm()` использовал `PendingIntent.getActivity(AlarmActivity)` — прямой запуск Activity из AlarmManager. На Android 12+ фоновый запуск Activity ограничен, и на многих устройствах (особенно китайских OEM) PendingIntent.getActivity молча не срабатывает.

**Исправление:** Будильники теперь используют `PendingIntent.getBroadcast(AlarmReceiver)` — ту же схему, что и напоминания/периодические. BroadcastReceiver гарантированно вызывается системой даже при убитом процессе. При получении события AlarmReceiver постит уведомление с `setFullScreenIntent(true)`, которое надёжно показывает AlarmActivity поверх блокировки.

- Удалён метод `buildAlarmPendingIntent()` (создавал PendingIntent к AlarmActivity)
- Удалён метод `cancelAlarm()` (целился в AlarmActivity; `cancel()` и так обслуживает все типы)
- `buildPendingIntent()` теперь принимает `isAlarm` флаг для маршрутизации в AlarmReceiver

### Баг 2: Сообщение будильника не появляется в чате

**Причина:** Несовместимость форматов дат в SQL-запросах. `scheduled_at` хранился в ISO 8601 (`2024-01-15T10:30:00.000Z`), а сравнение шло с `datetime('now')` SQLite (`2024-01-15 10:30:00`). Символ `T` (ASCII 84) всегда больше пробела (ASCII 32) — условие `scheduled_at <= datetime('now')` всегда ложно.

**Исправление:** Нормализация формата в запросе через `REPLACE(SUBSTR(scheduled_at, 1, 19), 'T', ' ')`:
- `getVisibleMessagesByChatId` — сообщения будильников/напоминаний корректно становятся видимыми после наступления времени
- `getMessagesForChatAtTime` — то же, с устранением параметра (используется `datetime('now')` напрямую)

### Улучшения надёжности

- **WakeLock** в AlarmActivity: экран/процессор не гаснут во время показа будильника
- **REQUEST_IGNORE_BATTERY_OPTIMIZATIONS** в манифесте: возможность отключить оптимизацию батареи
- **WAKE_LOCK** в манифесте: разрешение на удержание процессора

### Гайд по разрешениям для пользователя

При первом создании будильника показывается диалог с перечнем необходимых разрешений:
- Точные будильники (Android 12+)
- Отключение оптимизации батареи
- Показ поверх блокировки
- Автозапуск (Xiaomi, Huawei и др.)

## Изменённые файлы

- `android/app/src/main/java/com/lichka/AlarmScheduler.kt` — все типы уведомлений через BroadcastReceiver; удалены устаревшие методы
- `android/app/src/main/java/com/lichka/AlarmActivity.kt` — добавлен WakeLock для удержания экрана
- `android/app/src/main/AndroidManifest.xml` — добавлены WAKE_LOCK и REQUEST_IGNORE_BATTERY_OPTIMIZATIONS
- `src/entities/message/model/messageRepository.ts` — исправлено сравнение дат в SQL-запросах
- `src/widgets/message-composer/MessageComposer.tsx` — диалог-гайд по разрешениям перед созданием будильника
- `src/shared/config/locale.ts` — строки локализации для гайда по разрешениям (RU/EN)
- `src/entities/message/__tests__/messageRepository.test.ts` — обновлён тест под новый формат запроса

## Принятые решения

1. **BroadcastReceiver как единая точка входа** — все типы (будильник, напоминание, периодическое) используют PendingIntent.getBroadcast. Это гарантирует доставку при любом состоянии приложения.
2. **setFullScreenIntent вместо прямого запуска Activity** — уведомление + fullScreenIntent надёжнее прямого PendingIntent.getActivity на Android 12+.
3. **Нормализация дат на стороне SQL** — вместо изменения формата хранения (риск поломать существующие данные), преобразование делается в запросе через `REPLACE(SUBSTR(...))`.

## Известные ограничения

- На устройствах Xiaomi, Huawei, OPPO и др. может потребоваться ручное включение автозапуска в настройках системы
- `USE_FULL_SCREEN_INTENT` может быть отозван пользователем в настройках — fullScreenIntent не сработает (уведомление останется в шторке)
- Старые PendingIntent к AlarmActivity (созданные до этого обновления) не будут отменены — но они безопасно покажут AlarmActivity при срабатывании

- Ручное: создание будильника → блокировка экрана → проверка показа AlarmActivity
- Ручное: создание будильника → наступление времени → открытие чата → проверка видимости сообщения
- Unit-тесты: обновлён `getMessagesForChatAtTime` тест
- Не покрыто: E2E-тесты будильника (требуют манипуляции с системным временем)
