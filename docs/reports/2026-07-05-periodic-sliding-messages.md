# Исправление «скользящих» периодических сообщений

**Дата:** 2026-07-05
**Промпт/задача:** Исправить баг: периодические напоминания не создают «скользящее» сообщение в чате

## Что сделано

### Проблема
Периодическое сообщение (тип `periodic`) всегда отображалось в чате на позиции своего создания и никогда не «скользило» вперёд при срабатываниях. Шаблон (`type='periodic'`) был виден в чате всегда, а механизма создания/удаления экземпляров при fire не существовало.

### Решение
Разделение шаблона (только в Scheduled) и display-сообщения (в чате):

1. **`getVisibleMessagesByChatId()`** — исключил `periodic` из видимых в чате сообщений (шаблон больше не отображается как обычное сообщение)

2. **`getPeriodicDisplayMessages(chatId)`** — новая функция, генерирует синтетические display-сообщения:
   - Для каждого enabled periodic-шаблона вычисляет время последнего срабатывания: `createdAt + floor((now - createdAt) / interval) * interval`
   - Если срабатываний ещё не было (`fires === 0`) — не показывает
   - Возвращает ровно одно сообщение на шаблон (самое свежее) с ID `periodic:<templateId>`

3. **`ChatRoomScreen`** — обновлён:
   - `loadData()` объединяет обычные + периодические display-сообщения, сортирует по `createdAt`
   - Периодический рефреш каждые 30 секунд (пока чат открыт)
   - Обработчики delete/edit адаптированы для синтетических ID (извлекают templateId)
   - Поиск по сообщениям находит периодические шаблоны и скроллит к display-сообщению
   - Навигация по `messageId` (из Scheduled/нотификаций) учитывает синтетические ID

### Архитектурное решение
Выбран подход с **синтетическими сообщениями** (без изменения схемы БД):
- Шаблон остаётся в БД как `type='periodic'` — blueprint
- Display-сообщение вычисляется на лету при каждой загрузке
- При каждом срабатывании `createdAt` display-сообщения меняется → FlatList пересортировывает

## Изменённые файлы

- `src/entities/message/index.ts` — экспорт новых функций
- `src/pages/chat-room/ChatRoomScreen.tsx` — интеграция display-сообщений, polling, адаптация delete/edit/search
- `src/entities/message/__tests__/messageRepository.test.ts` — тесты для новых функций и обновлённой логики

## Принятые решения
- **Синтетические сообщения вместо отдельных DB-строк** — не требует миграции, проще в реализации
- **ID `periodic:<templateId>`** — консистентный ключ для FlatList; `createdAt` меняется в данных, но ключ стабилен
- **30-секундный polling** — компромисс между реактивностью и нагрузкой; native AlarmReceiver не может напрямую обновлять JS-сторону

## Известные ограничения
- Сообщение «скользит» только при рефреше (до 30 секунд задержки в худшем случае)
- При первом открытии чата после долгого отсутствия — сообщение сразу на последней позиции (пропущенные «промежуточные» сообщения не видны — это ожидаемое поведение)
- Нет истории отдельных срабатываний (только последнее)
