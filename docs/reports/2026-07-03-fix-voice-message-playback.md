# Исправление воспроизведения голосовых сообщений

**Дата:** 2026-07-03
**Промпт/задача:** Исправить ошибку: записанное аудио-сообщение не воспроизводится (нет звука и анимации)

## Что сделано

- Найдена и исправлена корневая причина: при сохранении пути к файлу записи не отрезался префикс `file://`
- Добавлены кросс-платформенные параметры `AudioChannels` и `AudioSamplingRate` в конфиг записи (iOS-ключи игнорировались Kotlin-кодом)

## Изменённые файлы

- `src/widgets/message-composer/MessageComposer.tsx:209` — добавлен `.replace('file://', '')` перед вычислением `relativeUri`
- `src/features/voice-record/useVoiceRecorder.ts:66-72` — добавлены `AudioChannels: 1`, `AudioSamplingRate: 16000`

## Принятые решения

- **Причина:** `react-native-audio-recorder-player` v4.5.0 возвращает URI в формате `file:///data/.../file.m4a` (через `Uri.fromFile().toString()`). При вычислении относительного пути в `MessageComposer` префикс `file://` не отрезался, из-за чего получался путь `file://media/voice/uuid.m4a`. После `resolveMediaPath()` итоговый абсолютный путь становился `/data/.../file://media/voice/uuid.m4a` — несуществующий файл. `startPlayer()` падал молча.
- **Исправление:** `result.uri.replace('file://', '').replace(${DocumentDirectoryPath}/, '')` даёт корректный относительный путь `media/voice/uuid.m4a`.

## Известные ограничения

- При авто-остановке по достижении лимита 60 сек запись теряется без создания сообщения — это существующее поведение, не относится к данному багу.

## Тестирование

- Типчекинг пройден (ошибок в изменённых файлах нет)
- Требуется ручное тестирование на устройстве: запись → воспроизведение голосового сообщения
