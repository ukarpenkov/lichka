# Отправка изображений в чат

**Статус:** draft

## Описание проблемы

На данный момент сообщения могут содержать только текст или голосовые записи. Пользователь не может прикрепить к сообщению изображение — скриншот, фото документа или просто картинку. Это ограничивает функциональность чата, особенно в сценариях «заметки самому себе», где часто нужно сохранить визуальную информацию.

## Предлагаемое решение

Добавить пятый тип сообщения — `image`, позволяющий отправлять в чат одно изображение на сообщение. Изображение может быть как самостоятельным (без текста), так и с подписью снизу.

### Пользовательский сценарий

1. Пользователь нажимает кнопку скрепки (Paperclip) в `MessageComposer`, справа от TextInput
2. Открывается системный пикер изображений (`react-native-image-picker`) — выбор из галереи или камера
3. Выбранное изображение проходит сжатие и показывается в виде превью в MessageComposer (заменяет поле ввода на время)
4. Пользователь может добавить подпись (текст снизу превью) и нажать «Отправить»
5. Изображение отображается в `MessageBubble`: картинка сверху, текст (если есть) снизу, meta-строка как обычно

### Правила

| Правило | Значение |
|---------|----------|
| Одно сообщение — одна картинка | `type: 'image'`, изображение хранится как файл, путь — в `payload` |
| Картинка без текста | `body` = `''`, только изображение |
| Картинка + текст | Текст отображается снизу картинки |

## Техническое решение

### Слой `entities/message`

**Изменение типа `MessageType`** (`src/entities/message/model/types.ts:1`):

```typescript
export type MessageType = 'simple' | 'reminder' | 'alarm' | 'periodic' | 'image';
```

**Расширение `Message` не требуется** — все необходимые поля уже есть:
- `type: 'image'` — идентифицирует сообщение-картинку
- `body: string` — опциональная подпись (пустая строка если без текста)
- `payload: string | null` — JSON с путём к файлу (`{ "uri": "media/images/<id>.jpg" }`)

Это повторяет паттерн голосовых сообщений (`type: 'simple'` + `payload.uri`).

### Слой `entities/message` — репозиторий

**Миграция БД** (`src/shared/db/db.ts` — миграция 6):

```sql
-- Обновляем CHECK constraint, SQLite не поддерживает DROP CONSTRAINT,
-- поэтому создаём новую таблицу и переносим данные
CREATE TABLE messages_new (
  id TEXT PRIMARY KEY NOT NULL,
  chat_id TEXT NOT NULL,
  type TEXT NOT NULL CHECK(type IN ('simple', 'reminder', 'alarm', 'periodic', 'image')),
  body TEXT NOT NULL DEFAULT '',
  scheduled_at TEXT,
  interval_minutes INTEGER,
  enabled INTEGER,
  payload TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  FOREIGN KEY (chat_id) REFERENCES chats(id) ON DELETE CASCADE
);

INSERT INTO messages_new SELECT * FROM messages;
DROP TABLE messages;
ALTER TABLE messages_new RENAME TO messages;
```

**Изменения в `messageRepository.ts`:**
- `createMessage` — без изменений (уже принимает произвольный `type`)
- `deleteMessage` — добавить удаление файла изображения по `payload.uri` (аналогично голосовым)
- `getVisibleMessagesByChatId` — добавить `'image'` в список видимых типов (рядом с `'simple'`)

## Альтернативы

### 1. Хранить картинки как base64 в `body`
**Отклонено:** body используется для текста и индексируется FTS. Base64-строка раздует FTS индекс. Поле `payload` уже существует и используется для голосовых — логично использовать его же.

### 2. Использовать отдельную таблицу `attachments`
**Отклонено:** Усложняет схему ради одного поля. Правило «одно сообщение = одна картинка» не требует нормализации. Добавление `payload` + `type: 'image'` полностью покрывает потребность без новой таблицы.

### 3. Разрешить несколько картинок в одном сообщении
**Отклонено:** Усложняет UI (галерея в баббле), хранение (массив URI в payload), и не соответствует текущему требованию «одно сообщение = одна картинка».

### 4. Fullscreen-просмотр при тапе
**Отложено до v2.** Требует модального просмотрщика с зумом и жестами. В v1 достаточно показа картинки в баббле.

## Оценка сложности

| Компонент | Часы |
|-----------|------|
| Миграция БД | 0.5 |
| `imageCompress.ts` | 1 |
| `mediaPath.ts` (saveImage) | 0.5 |
| `MessageComposer` (кнопка + превью + отправка) | 3 |
| `ImageMessage` widget | 2 |
| `MessageBubble` (интеграция) | 1 |
| Локализация (RU/EN) | 0.5 |
| Экспорт/импорт (включить изображения) | 1 |
| Тесты (≥80%) | 3 |
| **Итого** | **~12.5 часов** |

**Риски:**
- **Средний:** Зависимость от `react-native-image-picker` — уже используется в `ChatForm.tsx`, стабилен
- **Низкий:** Миграция с пересозданием таблицы — выполняется один раз, атомарна (транзакция)
- **Средний:** Размер БД при активном использовании картинок — сжатие JPEG 0.75 с 1920px должно держать ~200-500KB на изображение
- **Низкий:** Удаление файлов при удалении сообщения — паттерн уже отработан для голосовых

## Вопросы к ТЗ

1. **Полноэкранный просмотр.** Нужен ли тап по картинке для открытия на весь экран в v1, или откладываем? Влияет на объём работы и UX.

2. **Камера.** Нужен ли вариант «снять фото» прямо из приложения (а не только выбор из галереи)? `react-native-image-picker` поддерживает оба варианта.

3. **Максимальный размер.** Устраивает ли лимит 1920×1920px / JPEG 0.75? Или нужны другие значения? Это влияет на размер базы и скорость отображения.

4. **Превью перед отправкой.** Должно ли превью показываться в том же MessageComposer (заменяя поле ввода) или в отдельной модалке/боттомшите? Второй вариант даёт больше места для превью, но усложняет поток.

5. **Скрепка или плюс?** Иконка Paperclip (📎) или Plus (➕) для кнопки добавления? Paperclip более узнаваема для «прикрепить», Plus — для «добавить».

6. **Экспорт/импорт с картинками.** При экспорте в JSON включать изображения как base64 прямо в payload, или экспортировать отдельным zip-архивом (файлы + JSON)? Второе надёжнее для больших данных.

7. **Бэкап в Google Drive.** Нужно ли включать изображения в бэкап (сейчас бэкап — только JSON с текстовыми данными)? Файлы изображений могут значительно увеличить размер бэкапа.

8. **Ограничение на количество.** Нужен ли лимит на количество картинок в чате или на общий размер директории `media/images/`? Без лимита при активном использовании может закончиться место на устройстве.

9. **Редактирование.** Можно ли редактировать подпись к картинке после отправки? (Сейчас `edit` затрагивает только `body`, что автоматически даёт возможность менять подпись.)

10. **Совместимость с scheduled.** Нужна ли возможность запланировать отправку картинки (как reminder/alarm/periodic)? В текущем предложении `image`-тип и scheduled-типы разделены.
